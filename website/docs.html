HolyC Documentation

Full reference for the HolyC compiler — a C++17 / LLVM 18 implementation of Terry Davis's HolyC language.

1. Getting Started

Prerequisites

  • CMake 3.20+
  • LLVM 14–18 development libraries (llvm-dev, libclang-dev or equivalent)
  • C++17-compatible compiler — GCC 9+ or Clang 10+

Build from Source

git clone https://github.com/pacakutekagudabasa/HolyC.git
cd HolyC
mkdir build && cd build
cmake ..
make -j$(nproc)

Verify Installation

./build/hcc --help

You should see the list of available command-line flags printed to stdout.

Run the Test Suite

ASAN_OPTIONS=detect_leaks=0 bash tests/run_tests.sh

352 tests across 126 test files (interpreter, JIT, AOT, and formatter modes).

Usage Examples

# Interpret a program
./hcc --interpret hello.HC

# JIT compile and run
./hcc --jit hello.HC

# AOT compile to native binary
./hcc -o hello hello.HC
./hello

# AOT with optimizations and debug info
./hcc -O2 -g -o hello hello.HC

# Format source in-place
./hcc --format messy.HC

# Cross-compile for aarch64
./hcc --target aarch64-linux-gnu -o hello_arm hello.HC

3. Language Reference

HolyC is the programming language created by Terry A. Davis for TempleOS. This compiler implements the full language with extensions for modern systems.

3.1 Types

HolyC provides a complete set of fixed-width integer types, floating-point types, and a boolean.

TypeSizeDescription
I00Signed void (0 bytes)
U00Void type — used for functions that return nothing
U81 byteUnsigned 8-bit integer; also used for characters and strings
U162 bytesUnsigned 16-bit integer
U324 bytesUnsigned 32-bit integer
U648 bytesUnsigned 64-bit integer
I81 byteSigned 8-bit integer
I162 bytesSigned 16-bit integer
I324 bytesSigned 32-bit integer
I648 bytesSigned 64-bit integer — the default integer type
F324 bytes32-bit IEEE 754 floating point
F648 bytes64-bit IEEE 754 floating point — the default float type
Bool1 byteBoolean — TRUE or FALSE

Intrinsic Union Types

HolyC provides intrinsic union variants of all base types, used for type punning in TempleOS. These share the same size as their base type but allow reinterpreting the underlying bits.

TypeDescription
U8i, I8iIntrinsic union variants of U8 / I8
U16i, I16iIntrinsic union variants of U16 / I16
U32i, I32iIntrinsic union variants of U32 / I32
U64i, I64iIntrinsic union variants of U64 / I64
F64iIntrinsic union variant of F64

Pointer Types

Pointers are declared with * after the base type. Multi-level pointers are supported.

I64 *p;       // pointer to I64
I64 **pp;     // pointer to pointer to I64
U8 *str;      // string (pointer to U8)

Array Types

Arrays use bracket syntax with a compile-time size. Multi-dimensional arrays and initializer lists are supported.

I64 arr[10];                        // 10-element array
I64 m[3][4];                        // 3x4 matrix
I64 a[4] = {10, 20, 30, 40};       // initialized array
U8 s[8] = "hello";                 // string-initialized array

3.3 Control Flow

if / else

if (x > 0) {
  "Positive\n";
} else if (x == 0) {
  "Zero\n";
} else {
  "Negative\n";
}

for loop

I64 i;
for (i = 0; i < 10; i++)
  "i = %d\n", i;

while / do-while

while (x > 0) x--;

do {
  x++;
} while (x < 10);

switch / case

HolyC supports the standard switch/case statement as well as the case N..M: range syntax for matching a contiguous range of values.

switch (ch) {
  case 'a'..'z':
    "lowercase\n";
    break;
  case 'A'..'Z':
    "uppercase\n";
    break;
  case '0'..'9':
    "digit\n";
    break;
  default:
    "other\n";
}

goto

Labels and goto are fully supported, including backward jumps for loop-like patterns.

  goto done;
  "skipped\n";
done:
  "reached\n";

break / continue

break exits the innermost loop or switch. continue jumps to the next iteration of the innermost loop. Both behave as in standard C.

3.4 Functions

Basic Functions

I64 Add(I64 a, I64 b) {
  return a + b;
}

Default Parameters

Parameters may have default values. Callers can omit trailing arguments that have defaults.

U0 Greet(U8 *name = "World") {
  "Hello %s\n", name;
}

Variadic Functions (three-dot syntax)

Functions can accept a variable number of arguments using the ... syntax. Inside the function body, variadic arguments are accessed through the argv mechanism.

U0 PrintAll(I64 count, ...) {
  I64 i;
  for (i = 0; i < count; i++)
    "%d\n", argv[i];
}

Note: argv is a special implicit array available inside variadic functions that contains the variable arguments. It is indexed starting at 0, where argv[0] is the first variadic argument (after any fixed parameters).

Forward Declarations

Functions can be forward-declared by providing the signature without a body, terminated by a semicolon.

I64 Fib(I64 n);

3.5 Classes

Basic Classes

class Point {
  I64 x, y;
};

Inheritance

Single inheritance is supported with a colon after the class name.

class Point3D : Point {
  I64 z;
};

Methods

Methods are defined inside the class body. The this pointer is available implicitly.

class Vehicle {
  I64 speed;
  U0 SetSpeed(I64 s) {
    this->speed = s;
  };
  I64 GetSpeed() {
    return speed;
  };
};

Usage

Vehicle v;
v.SetSpeed(60);
"Speed: %d\n", v.GetSpeed();

Forward Declarations

Classes can be forward-declared for use in pointer types before the full definition is available.

class Node;

Keywords

  • lastclass — refers to the most recently defined class type, useful for self-referential patterns.
  • public — can precede class fields to indicate public accessibility.
// lastclass refers to the most recently declared class type
class Point { I64 x, y; };
lastclass p;  // equivalent to: Point p;
p.x = 10;

3.6 String Output

One of HolyC's most distinctive features is its string output syntax. A string literal used as a statement is automatically printed to stdout. Format specifiers and arguments follow with commas, just like printf — but without a function name.

// Simple string output (no function call needed)
"Hello World\n";

// With format specifiers
"Value: %d\n", 42;
"Name: %s, Age: %d\n", name, age;

Format Specifiers

%xHexadecimal (lowercase)

%BBinary (uppercase variant, same as %b)

Format Modifiers

Width, precision, and flags work as in C:

"%10d", val;          // right-aligned, width 10
"%-20s", name;        // left-aligned, width 20
"%08x", hex_val;      // zero-padded hex
"%.2f", pi;           // 2 decimal places
"%+d", signed_val;    // show sign for positive numbers

Available flags: - (left-align), + (force sign), 0 (zero-pad), (space before positive).

3.7 Inline Assembly

HolyC supports inline assembly for direct hardware access. Both simple and GCC-style extended syntax are available.

// Basic inline asm
asm("nop");

// Extended with GCC constraints
I64 result;
asm("mov %1, %0" : "=r"(result) : "r"(42));

3.8 Compile-Time Execution

The #exe directive executes HolyC code at compile time. Output generated via Emit is injected into the source stream as if it had been typed by the programmer.

#exe {
  I64 i;
  for (i = 0; i < 5; i++)
    Emit(MStrPrint("I64 var%d = %d;\n", i, i * 10));
}
// Generates at compile time:
//   I64 var0 = 0;
//   I64 var1 = 10;
//   I64 var2 = 20;
//   I64 var3 = 30;
//   I64 var4 = 40;

Note: Emit takes a single string. Use MStrPrint to format the string before passing to Emit.

3.9 Exception Handling

HolyC provides try/catch/throw for exception handling.

try {
  I64 *p = NULL;
  *p = 42;  // would crash
} catch {
  "Caught exception!\n";
}

throw;  // re-throw or raise an exception

3.10 Other Features

  • reg keyword — Register variable hint. Silently parsed and accepted for TempleOS compatibility but does not affect code generation.
  • lock {} blocks — Parsed as compound statements. Used in TempleOS for atomic sections.
  • typeof(expr) — Type specifier that resolves to the type of the given expression at compile time.
  • Multi-variable declarations — Multiple variables of the same type in one statement: 
    I64 i, j, k;
  • Adjacent string literal concatenation — Adjacent string literals are automatically joined at compile time: 
    "Hello " "World"   // equivalent to "Hello World"
  • extern "C" — Import C functions from system shared libraries for interoperability: 
    extern "C" I64 puts(U8 *s);
puts("Hello from libc");

Enums

// Basic enum (values start at 0)
enum Color {
  RED,
  GREEN,
  BLUE
};

// Enum with explicit values
enum Flags {
  FLAG_A = 1,
  FLAG_B = 2,
  FLAG_C = 4
};

Color c = RED;
"Color: %d\n", c;  // 0

Unions

union IntOrFloat {
  I64 i;
  F64 f;
};

union IntOrFloat u;
u.i = 42;
"As int: %d\n", u.i;

Typedefs

typedef I64 Size;
typedef U8 *String;

Size n = 100;
String s = "hello";

Function Pointers

I64 Add(I64 a, I64 b) { return a + b; }
I64 Mul(I64 a, I64 b) { return a * b; }

// C-style function pointer syntax
I64 (*op)(I64, I64) = &Add;
"Result: %d\n", op(3, 4);  // 7

op = &Mul;
"Result: %d\n", op(3, 4);  // 12

Static Variables

U0 Counter() {
  static I64 count = 0;
  count++;
  "Count: %d\n", count;
}

Counter;  // Count: 1
Counter;  // Count: 2
Counter;  // Count: 3

4. Preprocessor

The HolyC preprocessor runs before parsing and handles macro expansion, conditional compilation, and file inclusion.

Macros

Object-like and function-like macros are defined with #define and removed with #undef.

#define MAX_SIZE 1024
#define MIN(a, b) ((a) < (b) ? (a) : (b))
#undef MAX_SIZE

Variadic Macros

Macros can accept a variable number of arguments via ..., accessible as __VA_ARGS__.

#define LOG(fmt, ...) Print(fmt, __VA_ARGS__)

Token Paste and Stringification

The ## operator concatenates two tokens. The # operator converts a macro argument to a string literal.

#define CONCAT(a, b) a##b
#define STRINGIFY(x) #x

I64 CONCAT(my, Var) = 42;  // expands to: I64 myVar = 42;
"Name: %s\n", STRINGIFY(hello);  // expands to: "Name: %s\n", "hello";

Conditional Compilation

#ifdef DEBUG
  "Debug mode\n";
#endif

#ifndef RELEASE
  "Not release\n";
#endif

#if PLATFORM == 1
  "Platform 1\n";
#elif PLATFORM == 2
  "Platform 2\n";
#else
  "Unknown\n";
#endif

Mode-Specific Conditionals

HolyC provides directives to conditionally compile code based on the execution mode.

#ifaot
  // Only compiled in AOT mode
#endif

#ifjit
  // Only compiled in JIT mode
#endif

Other Directives

#include "file.HC"Include another source file. Search paths can be added with -I.
#assert exprCompile-time assertion. Compilation fails if the expression evaluates to false.
#exe { ... }Compile-time execution block (see Compile-Time Execution).

7. Architecture

The HolyC compiler is structured as a multi-stage pipeline. Each stage transforms the program representation and passes it to the next.

1. Lexer (src/lexer/)

The lexer (tokenizer) reads HolyC source text and produces a stream of TokenKind values. It recognizes all HolyC keywords, operators, identifiers, and literals (integer, floating-point, string, and character). Numeric parsing uses a 64-byte stack buffer to avoid heap allocation. The lexer feeds tokens to the preprocessor and then the parser.

2. Preprocessor (src/preprocessor/)

The preprocessor operates on the token stream before parsing. It handles:

  • Macro expansion — object-like and function-like macros
  • Conditional compilation — #if, #ifdef, #ifndef, #elif, #else, #endif, #ifaot, #ifjit
  • File inclusion — #include with search path support
  • Variadic macros — __VA_ARGS__
  • Token pasting — ## operator
  • Stringification — # operator
  • Compile-time execution — #exe { } blocks

The isActive() check that determines whether the current token stream is inside an active conditional branch runs in O(1) time via an inactive_depth_ counter, rather than scanning the condition stack.

3. Parser (src/parser/)

A recursive-descent parser builds an Abstract Syntax Tree (AST) from the token stream. It uses a circular lookahead buffer for O(1) token consumption. The parser handles all HolyC-specific syntax including string output statements, postfix type casts, the backtick power operator, and default argument values. All AST nodes are arena-allocated, avoiding individual new/delete calls. Class names are tracked in a set with transparent comparison to avoid unnecessary string allocations during identifier lookups.

4. Semantic Analysis (src/sema/)

The semantic analysis pass walks the AST to perform type checking, name resolution, and type inference. Primitive types (I64, U8, F64, etc.) are cached as singletons to avoid per-literal arena allocation. The pass uses NodeKind-based dispatch with static_cast instead of dynamic_cast chains for performance. It resolves class inheritance hierarchies, validates function call arguments against parameter lists, and ensures type safety for all operations.

5. Interpreter (src/interpreter/)

The tree-walk interpreter executes the AST directly using a frame-based call stack. Each Value is 32 bytes, using a custom SmallStr SSO type (16 bytes, inline for strings up to 14 characters). Over 100 built-in functions are dispatched via a hash map with a single-lookup pattern. The interpreter supports full HolyC semantics including goto (with backward jumps for loop patterns), try/catch/throw exception handling, string output, pointer arithmetic, and class method dispatch. Variables are scoped correctly with direct insertion into the current frame to prevent cross-frame overwrites.

6. LLVM Codegen (src/codegen/)

The code generator lowers the AST to LLVM IR. It features type memoization (caching Type* to llvm::Type* mappings), struct layout computation with field index optimization for O(1) field access, and proper handling of pointer arithmetic via GEP instructions. Cross-type conversions (pointer-to-int, int-to-pointer, float widening/narrowing) are handled at all boundary points. Vararg functions use a TLS buffer ABI for user-defined variadics.

7. JIT Engine

The JIT backend uses LLVM's ORC LLJIT engine. It includes a SHA-256 content-based bitcode cache: before code generation, the source file's content is hashed and checked against cached bitcode files. On a cache hit, the cached bitcode is loaded directly via parseBitcodeFile, bypassing parsing, semantic analysis, and code generation entirely. The cache key uses file content (not path), so stale cache entries are impossible.

8. AOT Compiler

The AOT backend compiles HolyC to native binaries via LLVM's machine code emission. It supports optimization levels -O0 through -O3, DWARF debug information (-g), and cross-compilation to aarch64-linux-gnu and x86_64-w64-mingw32 targets. The output can be either an object file (.o) or a fully linked executable.

9. Arena Allocator

All AST nodes are allocated from a custom arena allocator using 64KB blocks. The arena tracks destructors for non-trivially-destructible objects (those containing std::vector or std::string fields) and calls them in reverse order during cleanup. This eliminates the overhead of individual new/delete calls for the thousands of AST nodes created during compilation and ensures no memory leaks.

9. Testing

The HolyC compiler has 352 tests covering all three execution modes.

Test Suite Overview

  • 352 tests across 126 test files
  • Each test file is run in 3 execution modes: interpreter, JIT, and AOT
  • Additional formatter mode tests verify formatting round-trips
  • Test source files use the .HC extension and live in the tests/ directory
  • Expected output is stored in corresponding .expected files

Running the Tests

ASAN_OPTIONS=detect_leaks=0 bash tests/run_tests.sh

The ASAN_OPTIONS=detect_leaks=0 environment variable is needed because the test runner uses AddressSanitizer builds.

Test File Structure

Each test consists of a .HC source file and an .expected file containing the exact expected stdout output. The test runner compiles and executes the source file in each mode, then compares the actual output to the expected output. Any mismatch is reported as a failure.

Writing New Tests

To add a new test:

  1. Create tests/my_test.HC with the HolyC program.
  2. Create tests/my_test.expected with the exact expected output.
  3. Run the test suite to verify all three modes produce the expected output.

10. Security

The HolyC compiler has undergone multiple rounds of security auditing with all findings fixed. The following hardening measures are in place throughout the runtime and compiler:

Memory Safety

  • Bounds-checked array indexing — Array accesses validate that the index is non-negative and within the allocated size. Indices exceeding 0x10000000 / elemSize are rejected.
  • Buffer overflow protection — String builtins (StrCpyN, StrCatN, SPrint) check the destination buffer capacity via heap_.size() before writing and truncate if needed.
  • Memory operation size limits — MemMove, MemSet, and MemCpy cap their size argument at 500MB to prevent runaway operations.
  • Allocation guards — ReAlloc rejects negative or huge (>1GB) size requests. FileReadAll guards against files larger than 256MB.

Integer Safety

  • Shift overflow guards — All shift operations (<<, >>, <<=, >>=) validate that the shift amount is in the range [0, 63].
  • Division by zero and INT64_MIN protection — Division and modulo operations guard against INT64_MIN / -1 undefined behavior.
  • Bit operation guards — Bt, Bts, Btr, and Btc validate that the bit index is in [0, 63] and use 1ULL shifts. Null pointer parameters are checked.
  • Bit-field extraction safety — Bit-field operations validate that width and start position are within valid ranges.
  • Safe string-to-number parsing — Str2I64 and Str2F64 wrap their underlying C++ calls in try-catch blocks.

Data Structure Safety

  • Capacity caps — Dynamic data structures (CVec, CQueue, CStack) are capped at 250 million elements to prevent memory exhaustion.
  • Null allocation checks — HashMapRehash and HashMapInit verify that MAlloc/CAlloc return non-null before proceeding.

File System Safety

  • Path traversal prevention — The #include directive rejects paths containing .. components to prevent directory traversal attacks. System include paths (e.g. /usr/include/) are allowed for C interop.

Regex Safety

  • ReDoS protection — The regex engine enforces a maximum of 100,000 iterations per match alongside the existing recursion depth limit, preventing catastrophic backtracking on adversarial patterns.

Runtime Hardening

  • Null pointer guards — Pointer dereferences and bit-manipulation builtins check for null before access.
  • Format string width overflow — The formatPrint implementation caps width specifiers at 10,000 to prevent memory exhaustion.
  • GetStr cast safety — The max parameter is clamped to 1GB before casting to int.
